Where The Logic Should Be?

如果有这样的代码：


function Clip(pos:vec3, direction:vec3):void{/* your logic */}

function GoingToClipSomething():void{
    const _pos:vec3=new vec3(a,b,c);
    const _dir:vec3=new vec3(d,e,f);
    Clip(_pos,_dir);
}

function GoingToClipSomeOtherThing():void{
    //....
    Clip(...);
}

function GoingToClip...;

Clip 函数会在 pos 位置开始，将 direction 一侧的所有东西裁剪掉（想象剖切钢管这件事）。 上面的逻辑没有问题，不过需求有变，希望总是剖切视野能见到的一侧。那么就需要判断direction 的方向是否指向视野原点，如果不是就取反即可，也很简单，不是么。

但，思考点来了：这个逻辑我要添加在哪里？是 GoingToClip 函数里面，还是 Clip 函数里面？

  • 放在GoingToClip... 的话，Clip逻辑保持简单直接： 在pos位置开始，将direction 一侧的所有东西裁剪掉。
  • 放在Clip的话，调用Clip的众多函数保持不变，修改简单快捷。

那么怎么抉择？

单从软件设计的角度来说，Clip 作为被调用方，应该保持通用性、简单性， 这样才能服务于其他逻辑。

通用性体现在它被众多“裁剪”逻辑调用，要保持中立； 简单性体现在任务明确、依赖低。

如果新需求加在 Clip 中，就势必增加它的依赖性（依赖视野原点与 direction 的空间关系），从而降低封闭性，增加受干扰的机会。 另外，它也变得不再那么“通用”——如果有些地方就是希望按照原来的逻辑裁剪，又该怎么办？

更深一层来说，Clip 作为服务者，它的出度远远小于入度1。

这意味着 Clip 的变动会比较敏感，影响面较大。在程序设计中（包括架构层面对一个模块的定位）， 它属于“核心”代码，不应该轻易改动理论上也不会频繁变化的逻辑。

而处于外围的 GoingToClipSomething 等函数则相对善变， 需求变化、适配逻辑等都可能涉及它们，因此应该尽可能保持其较低的入度， 避免影响其他代码。

  1. 被一处逻辑调用算一个入度，自己调用其他逻辑算一个出度。
